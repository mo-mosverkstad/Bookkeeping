# Analysis Report: Visual-First Cheat Sheet System

## Executive Summary

The project began as an attempt to create a spreadsheet-like cheat sheet editor where cells contain a source language that automatically renders mathematics, diagrams, relationships, and visual structures. During development, several seemingly reasonable editor designs failed in practice, revealing important Human-Computer Interaction (HCI) constraints.

The final direction is a **visual-first knowledge system** rather than a traditional spreadsheet, note-taking application, or textbook editor. The central insight is that knowledge should be encoded primarily through visual relationships rather than prose. The source language serves as a construction mechanism rather than the primary artifact.

---

# 1. Core Philosophy

## Traditional Knowledge Systems

Most textbooks and note systems are:

```text
Source Text (Primary)

↓ supports

Tables
Equations
Diagrams
Images
```

Knowledge is encoded primarily in paragraphs.

Visual elements are supplementary.

---

## Proposed Cheat Sheet System

The proposed system reverses this relationship:

```text
Visual Structure (Primary)

↓ generated by

Source Language
```

Examples:

Instead of:

> The gradient points in the direction of maximum increase and is perpendicular to the level curve.

Represent:

```text
        ↑ ∇f
        │
        │
~~~~~~~~~~~~~
 level curve
```

with minimal annotation.

The visual relationship itself contains the information.

---

# 2. Design Goals

The system should maximize:

## Visual Encoding

Knowledge represented through:

* spatial relationships
* grouping
* color
* containment
* arrows
* adjacency
* diagrams
* equations

rather than long text descriptions.

---

## Stability

Users build spatial memory.

Examples:

```text
Blue section = Derivatives
Green section = Integrals
```

or

```text
Top-right = Optimization
```

Stable layouts allow rapid recall.

Therefore:

* no unexpected movement
* no layout shifts
* no wrapping reflows
* no floating instability

---

## Content Dominance

The content should occupy most of the screen.

The editor exists to support the content.

Not vice versa.

---

# 3. Findings From Failed Designs

## Version 1: Right Sidebar Editor

Layout:

```text
Navigation | Table | Source Editor
```

### Advantages

* Source always visible.
* Simple implementation.

### Problems

* Constant horizontal eye movement.
* Reduced table width.
* Source editor became narrow.
* Excessive text wrapping.
* Split attention.

### Conclusion

Rejected.

The sidebar consumed too much valuable horizontal space.

---

## Version 2: Floating Source Editor

Layout:

```text
Rendered Cell
Source Editor Above Cell
```

### Advantages

* Excellent locality.
* Minimal eye movement.
* Clear connection between source and output.

### Problems

Technical:

* Scroll synchronization.
* Panning issues.
* Clipping.
* Viewport edge cases.
* Ownership ambiguity.

Usability:

* Text could become unreachable.
* Large columns caused instability.
* Difficult to maintain.

### Conclusion

Conceptually attractive but practically fragile.

The implementation complexity outweighed the locality benefits.

---

## Popup Editors

Layout:

```text
Popup Window
```

### Advantages

* Large editing space.
* Easy implementation.

### Problems

* Loses neighboring context.
* Breaks adjacency.
* Increases cognitive load.
* User must remember previous location.

### Conclusion

Rejected.

Especially unsuitable for a visual knowledge system.

---

# 4. Key HCI Discovery

A major realization emerged:

## The Goal Is Not Zero Eye Movement

Initially the objective was:

* maximum locality
* minimum eye movement

However:

```text
Perfect locality
≠
Best usability
```

The floating editor achieved excellent locality but introduced instability.

A stable interface with slightly more eye movement is often superior.

---

## Stability Beats Perfect Locality

Tradeoff:

### Floating Editor

Pros:

```text
Excellent locality
```

Cons:

```text
Instability
Complexity
Synchronization issues
```

---

### Fixed Editor

Pros:

```text
Stable
Predictable
Simple
Scalable
```

Cons:

```text
More eye movement
```

Result:

Stability is more valuable.

---

# 5. Final Table Editor Design

## Layout

```text
[Undo] [Redo] [Open] [Save]

[Table Name]
[Tabs]

────────────────────────

Main Table Area

────────────────────────

Source Editor (~5 lines)
```

---

## Characteristics

### Minimal Top Bar

Only system-level operations:

* Undo
* Redo
* Open
* Save

No excessive chrome.

---

### Dynamic Title

The title changes with the current sheet.

However:

* fixed width
* stable placement

Reason:

Visual stability matters more than pixel optimization.

---

### Tabs

Located near the top.

Inspired by:

* Excel
* browser tabs

Reason:

Users build spatial memory.

Example:

> "Calculus is the third tab."

instead of:

> "Search for Calculus."

---

### Large Content Area

The table occupies the overwhelming majority of screen space.

This reflects the philosophy:

```text
Visual artifact = primary
```

---

### Bottom Source Editor

Approximately:

* 5 lines high
* small font
* fixed position

Purpose:

* edit source
* preview updates immediately

Advantages:

* stable
* scalable
* extensible

Future:

```text
[Source]
[AI]
[Search]
[Tools]
```

can share the same workspace.

---

# 6. No Table Wrapping

Proposal:

Never wrap table columns.

Example:

Bad:

```text
A B C
D E F
```

when window shrinks.

Good:

```text
A B C D E F
```

with scrolling.

Reason:

Wrapping destroys spatial predictability.

Users remember locations.

They do not remember adaptive rearrangements.

---

# 7. Diagram Editor Philosophy

The diagram editor differs fundamentally from the table editor.

---

## Table Mode

Source edits:

```text
Selected Cell
```

Model:

```text
Content
↓
Inspector
```

---

## Diagram Mode

Source describes:

```text
Entire Structure
```

Model:

```text
Source Structure | Diagram
```

The source is not an inspector.

It is the diagram's backbone.

---

## Functional Groups

Structure:

```text
Main Group
 ├─ A
 ├─ B
 │   ├─ B1
 │   └─ B2
 └─ C
```

Produces:

```text
Visual Diagram
```

The source itself is hierarchical.

Therefore a vertical source panel on the left is appropriate.

---

## Why Left Side Works Here

Unlike the spreadsheet:

```text
Source | Diagram
```

resembles:

```text
Code | Preview
```

Users constantly compare structure and output.

Horizontal alignment becomes beneficial.

---

# 8. Navigation Philosophy

Another major finding:

## Overlays Are Not Always Better

Modern interfaces frequently replace everything with:

```text
Dropdowns
Popups
Overlays
```

This often harms discoverability.

---

## Stable Navigation Preferred

Users develop spatial memory:

```text
Sheet A
Sheet B
Sheet C
```

Visible tabs support:

* recognition
* muscle memory
* orientation

Hidden menus require recall.

---

# 9. Keyboard and Mouse Philosophy

## Principle

Every important action should be accessible without a keyboard.

---

### Mouse

Provides:

```text
Discoverability
```

---

### Keyboard

Provides:

```text
Efficiency
```

---

Examples:

```text
Ctrl+E
```

Focus editor.

Equivalent:

```text
Click Editor Tab
```

---

The shortcut accelerates.

It does not replace.

---

# 10. Calculator Analogy

A useful analogy emerged.

Suppose a calculator adds:

```text
2nd
3rd
4th
```

modifiers to reuse keys.

The physical complexity decreases.

The cognitive complexity increases.

Similarly:

```text
Few controls
```

does not imply:

```text
Simple system
```

The true goal is:

```text
Minimize thinking
```

not:

```text
Minimize buttons
```

# 11. Overall Architectural Direction

The project is evolving away from:

* spreadsheet software
* note-taking software
* textbook authoring

and toward:

## Visual Knowledge Authoring System

Characteristics:

* visual-first
* source-second
* relationship-centric
* diagram-centric
* stable layout
* minimal chrome
* strong spatial memory

---

## Final Design Principles

### 1. Visual First

Visual relationships contain knowledge.

---

### 2. Source Second

Source language constructs visual knowledge.

---

### 3. Stability Over Cleverness

Avoid floating complexity.

Avoid adaptive rearrangement.

---

### 4. Spatial Memory Matters

Users remember locations.

Preserve them.

---

### 5. Content Dominates

UI chrome should remain minimal.

---

### 6. Different Content Types Deserve Different Layouts

Tables:

```text
Content
↓
Source
```

Diagrams:

```text
Source | Diagram
```

---

### 7. Keyboard Accelerates

Keyboard shortcuts improve speed.

They do not define functionality.

---

## Final Conclusion

The most important discovery is that this project is not fundamentally a spreadsheet with rendering capabilities. It is a **visual knowledge system** where the rendered relationships are the primary artifact and the source language is merely the mechanism used to create them.

This realization explains many earlier design failures. Sidebar editors, floating editors, and popup editors were all attempting to optimize the editing mechanism. The successful designs instead optimize the visual artifact itself. Once the content is treated as the primary object and the source as a supporting tool, the interface naturally converges toward a stable, content-dominant workspace with minimal chrome, strong spatial consistency, and layouts tailored to the structure of the knowledge being represented.
